# Contributing to markview

Thanks for considering a contribution. markview is small by design — single Python file, no build step, no framework. That shapes what fits and what doesn't.

## Project goals

1. **Stay small and fast.** Cold start under a second. No daemons, no trays, no sync.
2. **Native feel.** GTK3 widgets, system theme, keyboard-first. No custom window chrome.
3. **No Electron, no server.** A document editor should not need a Chromium process.
4. **Optional online.** Features that require the network (KaTeX, Mermaid CDN, pandoc when installed) must degrade gracefully when offline or uninstalled.

If your idea fits, let's ship it.

## Before you start

1. Look at [open issues](https://github.com/Canary-Builds/markview/issues) — the item may already be in progress.
2. Check [ROADMAP.md](ROADMAP.md) — your feature may already be scoped to a future release.
3. For a non-trivial change, **open an issue first** and sketch the approach. Saves both of us time.

## Development setup

```bash
sudo apt install python3-gi gir1.2-gtk-3.0 gir1.2-webkit2-4.1 \
                 gir1.2-gtksource-4 python3-markdown python3-pygments

git clone https://github.com/<you>/markview.git
cd markview
python3 markview.py README.md
```

Optional:

```bash
sudo apt install python3-html2text pandoc     # extra features
python3 -m pip install --user ruff pyflakes    # linters
```

No virtualenv is needed — the GI bindings are system packages.

## Code style

- **Python ≥ 3.10.** Use modern syntax (`str | None`, `list[dict]`, etc.).
- Run `ruff check markview.py` before submitting. Zero warnings preferred.
- Keep `markview.py` as a single file. Modules can be added if the line count grows past ~2500.
- Comments are for *why*, not *what*. No autogenerated docstrings.
- Match existing naming: private methods `_camel_case`, helpers module-level.
- Defer cost: debounce timers for expensive work (see `LIVE_PREVIEW_DEBOUNCE_MS`, `WORD_COUNT_DEBOUNCE_MS`, etc.).
- Every new feature that requires activation should have a keyboard shortcut AND a palette action. Toolbar additions are reserved for core editing primitives.

## Test before you send

There is no automated GUI test suite; expected manual verification:

1. `python3 -m py_compile markview.py` — no syntax errors.
2. Launch with a file, exercise any widget you touched.
3. Exercise the golden path for at least three features your change could regress (e.g. save/reload, split view, palette).
4. Confirm there are no Python warnings, `Gtk-CRITICAL`, or unhandled GLib errors on stderr.

A smoke test script is welcome if you add one — see `docs/wiki/Development.md`.

## Commits

- Conventional, present tense, imperative (`Add X`, `Fix Y`).
- One logical change per commit.
- Reference the issue ID in the body: `Refs #42`.
- For user-visible changes, update `CHANGELOG.md` under `[Unreleased]`.

## Pull requests

- Branch from `main`.
- Keep PRs focused — one feature, one bug fix.
- Fill in the PR template.
- If behaviour changes, update the README and relevant wiki page in the same PR.

## Releases

- Maintainer bumps `__version__` in `markview.py`.
- CHANGELOG section moves from `[Unreleased]` to the new version heading.
- Tag the commit: `git tag -a vX.Y.Z -m 'vX.Y.Z — summary'`.
- Push `main` and the tag: `git push origin main --tags`.

## Code of Conduct

This project follows the [Contributor Covenant](CODE_OF_CONDUCT.md). Be kind, stay on topic, and assume good faith.
